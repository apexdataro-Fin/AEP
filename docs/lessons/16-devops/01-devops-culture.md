# هندسة DevOps

> **"DevOps ليست وظيفة. ليست أداة. إنها ثقافة تعاون بين التطوير والعمليات — معاً نحو منتج أفضل."**

## 🎯 أهداف التعلم

بعد إكمال هذا الدرس، ستكون قادراً على:
- شرح إطار CALMS وتطبيقه في مؤسسة حقيقية
- قياس أداء الفريق باستخدام DORA metrics
- كتابة Postmortem احترافي Blameless
- تطبيق مفاهيم SRE و Error Budgets
- قيادة تحول DevOps في مؤسستك

## ١. ما هو DevOps حقاً؟

### المشكلة القديمة — جدار الارتباك

```
┌─────────────┐         ┌─────────────┐
│  تطوير      │         │  عمليات     │
│  "الكود     │  ────→  │  "الكود لا  │
│   جاهز!"    │  ←────  │   يشتغل!"   │
└─────────────┘         └─────────────┘
         ↕                    ↕
    يلوم العمليات         يلوم التطوير
         ↕                    ↕
         └──── المنتج يتأخر ────┘
```

حل DevOps: **"أنت تبنيها، أنت تشغلها."** نفس الفريق يطور وينشر ويدير. لا يوجد "نحن" و"هم".

### 🟢 التفسير البسيط

تخيل مطعماً حيث الطباخ يطبخ ثم يرمي الطعام من النافذة للنادل. DevOps هو أن يجلس الطباخ والنادل معاً ويفكران: كيف نجعل التجربة أفضل للزبون؟

## ٢. إطار CALMS — بعمق

| العنصر | المعنى | مثال CloudNova | كيف تقيسه؟ |
|--------|--------|----------------|------------|
| **C**ulture | مسؤولية مشتركة، لا لوم | المطورون تحت الطلب لخدماتهم | عدد postmortems blameless |
| **A**utomation | CI/CD، IaC، اختبارات | GitHub Actions + Terraform | نسبة النشر الآلي |
| **L**ean | دفعات صغيرة، إزالة الهدر | فروع قصيرة، نشر متكرر | Lead time for changes |
| **M**easurement | مقاييس، مراقبة | DORA metrics | النخبة/متوسط/متدني |
| **S**haring | توثيق، نقل معرفة | Postmortems، Pair Programming | عدد المراجعات |

### 🟣 تطبيق CALMS في CloudNova — القصة الحقيقية

**قبل DevOps:**
- فريق تطوير (١٢ شخصاً) وفريق عمليات (٤ أشخاص)
- نشر مرة كل أسبوعين — يوم الخميس ٢ صباحاً (خوفاً من المشاكل)
- ٤٠٪ من النشرات تفشل وتُسترجع
- معدل MTTR = ٤ ساعات

**التحول — شهراً بشهر:**

| الشهر | النشاط | النتيجة |
|-------|--------|---------|
| ١ | دمج مهندس عمليات في كل فريق تطوير | قلل سوء الفهم ٦٠٪ |
| ٢ | أتمتة CI/CD (GitHub Actions) | وقت النشر: ساعتين ← ١٢ دقيقة |
| ٣ | أول Blameless Postmortem | توقف اللوم — بدأ التعلم |
| ٤ | بدء Pair Programming بين Dev و Ops | نقل معرفة في الاتجاهين |
| ٥ | قياس DORA baseline | اكتشفنا أننا "متوسطون" |
| ٦-٩ | تحسين كل مقياس | انتقلنا من متوسط إلى نخبة في ٣ مقاييس |
| ١٢ | فريق DevOps يتحول لـ Platform Engineering | بنينا منصة داخلية |

## ٣. DORA Metrics — كيف تقيس أداءك؟

| المقياس | النخبة | المتوسط | المتدني | CloudNova (الآن) |
|---------|--------|---------|---------|-----------------|
| **تكرار النشر** | حسب الطلب | أسبوعي | شهري | ٦/يوم ✅ |
| **Lead Time** | < ساعة | أسبوع | > شهر | ٢.٥ ساعة ✅ |
| **MTTR** | < ساعة | يوم | > شهر | ٣٥ دقيقة ✅ |
| **Change Failure Rate** | < ٥٪ | ١٠-١٥٪ | > ١٥٪ | ٢.٨٪ ✅ |

### كيف تحسب هذه المقاييس؟

```python
# لوحة قيادة DORA — CloudNova
from datetime import datetime, timedelta

class DORAMetrics:
    def deploy_frequency(self, deployments_this_week):
        return f"{deployments_this_week / 7:.1f}/day"
    
    def lead_time_hours(self, commit_time, deploy_time):
        """من commit إلى إنتاج"""
        return (deploy_time - commit_time).total_seconds() / 3600
    
    def mttr_minutes(self, incidents):
        """متوسط وقت استعادة الخدمة"""
        total = sum(i.restored_at - i.detected_at for i in incidents)
        return (total.total_seconds() / len(incidents)) / 60
    
    def change_failure_pct(self, deployments, failures):
        return (failures / deployments) * 100

metrics = DORAMetrics()
print(f"Deploy Frequency: {metrics.deploy_frequency(42)}")     # 6.0/day
print(f"Lead Time: {metrics.lead_time_hours(commit, deploy)}h") # 2.5h
print(f"MTTR: {metrics.mttr_minutes(incidents)}min")            # 35min
print(f"CFR: {metrics.change_failure_pct(250, 7)}%")            # 2.8%
```

## ٤. ثقافة Postmortem الـ Blameless

> **"لا نبحث عن المخطئ. نبحث عن الخطأ — وكيف نمنعه."**

### 🚨 حادثة CloudNova: فقدان بيانات جلسات المستخدمين

```markdown
# Postmortem: Redis Cluster Failure — ١٥ مارس ٢٠٢٤

## 📊 ملخص
- **المدة:** ٣٧ دقيقة (١٠:١٣ - ١٠:٥٠)
- **التأثير:** ١٢,٠٠٠ مستخدم فقدوا جلساتهم (تسجيل خروج مفاجئ)
- **السبب الجذري:** Redis cluster split بسبب خطأ في network partition handling

## ⏱️ الجدول الزمني
| الوقت | الحدث |
|-------|-------|
| 10:13 | Redis node-3 يفقد الاتصال بـ node-1 |
| 10:13 | Cluster يدخل split-brain — كلا الجانبين يعتقد أنه primary |
| 10:15 | أول بلاغ من دعم العملاء: "مستخدمون يخرجون فجأة" |
| 10:18 | PagerDuty ينبه (Redis memory usage غير طبيعي) |
| 10:22 | مهندس on-call يبدأ التحقيق |
| 10:31 | اكتشاف split-brain |
| 10:35 | إيقاف node-3 — cluster يستعيد حالته |
| 10:42 | الجلسات تبدأ بالعودة تدريجياً |
| 10:50 | عودة كاملة — ٣٧ دقيقة تعطل |

## 🔍 الأسباب الجذرية (5 Why's)
1. لماذا دخل Redis split-brain؟ ← network partition + cluster-require-full-coverage = no
2. لماذا لم يتعامل cluster مع partition؟ ← إعدادات غير مناسبة (quorum=1)
3. لماذا الإعدادات غير مناسبة؟ ← استخدمنا defaults دون مراجعة
4. لماذا لم نراجع defaults؟ ← لا توجد عملية مراجعة إعدادات لقواعد البيانات
5. لماذا لا توجد عملية مراجعة؟ ← لم نوثق متطلبات الإنتاج

## ✅ الإجراءات التصحيحية
| # | الإجراء | المسؤول | الموعد |
|---|---------|---------|--------|
| 1 | رفع quorum إلى N/2+1 | SRE | فوراً |
| 2 | إضافة integration test لـ split-brain | Dev | هذا الأسبوع |
| 3 | إنشاء runbook لـ Redis cluster failure | SRE | هذا الأسبوع |
| 4 | مراجعة إعدادات كل قواعد البيانات | SRE + Dev | هذا الشهر |
| 5 | إضافة alert على network partition | Monitoring | هذا الأسبوع |
```

### 📝 عناصر الـ Postmortem الجيد

1. **ملخص** — ماذا حدث في جملة واحدة؟
2. **الجدول الزمني** — دقيقة بدقيقة (لا تلوم أحداً)
3. **التأثير** — كم مستخدم؟ كم خسارة؟
4. **السبب الجذري** — 5 Why's للوصول للجذر
5. **الإجراءات** — محددة، بمسؤول وموعد
6. **الدروس** — ماذا تعلمنا للمستقبل؟

## ٥. SRE — Site Reliability Engineering

| المفهوم | المعنى | مثال CloudNova |
|---------|--------|---------------|
| **SLO** | Service Level Objective | 99.9% توفر شهرياً |
| **SLI** | Service Level Indicator | القياس الفعلي: 99.95% |
| **SLA** | Service Level Agreement | ما وعدت به العميل (قد يكون أقل من SLO) |
| **Error Budget** | 1 - SLO = مسموح لك بـ X دقيقة تعطل | 0.1% = ٤٣ دقيقة/شهر |

### 🟣 Error Budget — الرياضيات وراء القرارات

```
SLO = 99.9% → Error Budget = 0.1%
في شهر = ٣٠ يوم × ٢٤ ساعة × ٦٠ دقيقة = ٤٣,٢٠٠ دقيقة
Error Budget = ٠.١٪ × ٤٣,٢٠٠ = ٤٣.٢ دقيقة تعطل مسموح بها شهرياً

سيناريو:
- ١-١٥ مارس: ٣ نشرات، ٠ دقائق تعطل ← الميزانية ممتلئة
- ١٦ مارس: تعطل ٢٠ دقيقة ← الميزانية: ٢٣.٢ دقيقة متبقية
- ٢٠ مارس: تعطل ١٥ دقيقة ← الميزانية: ٨.٢ دقيقة متبقية ⚠️
- القرار: أوقف كل النشرات الجديدة! ركز على الاستقرار.

لو استهلكت الميزانية كاملة ← انتهاك SLO ← تعويضات للعملاء (SLA)
```

## ٦. أنماط DevOps الخاطئة — واحذر منها!

| النمط | المشكلة | الحل |
|-------|---------|------|
| "فريق DevOps منفصل" | أنشأت صومعة جديدة بدل كسر القديمة | ادمج مهندسي العمليات في فرق التطوير |
| "اشترينا Kubernetes" | الخلط بين الأداة والفلسفة | DevOps ثقافة. Kubernetes مجرد أداة |
| "نرمي للعمليات" | لا ملكية مشتركة | المطورون on-call لخدماتهم |
| "التسليم السريع بأي ثمن" | تجاهل الجودة والاستقرار | السرعة + الاستقرار = توازن |
| "DevOps = أتمتة فقط" | تجاهل الثقافة والقياس والمشاركة | CALMS كاملة — وليس A فقط |

## 🧠 أسئلة للمراجعة النشطة (Active Recall)

1. اشرح CALMS لزميل جديد — أعط مثالاً لكل حرف من CloudNova.
2. ما الفرق بين SLO و SLA؟ لماذا يكون SLO عادة أعلى من SLA؟
3. متى توقف النشرات بسبب Error Budget؟ أعط مثالاً رقمياً.
4. لماذا "فريق DevOps منفصل" نمط خاطئ؟

## ✍️ تمرين Feynman

اشرح مفهوم Error Budget لمدير غير تقني. استخدم تشبيه الميزانية المالية. لماذا من المنطقي التوقف عن النشرات الجديدة عندما "تنفد الميزانية"؟
