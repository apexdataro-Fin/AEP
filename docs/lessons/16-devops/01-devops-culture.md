# هندسة DevOps

> **"DevOps ليست وظيفة. ليست أداة. إنها ثقافة تعاون بين التطوير والعمليات — معاً نحو منتج أفضل."**

## ما هو DevOps حقاً؟

المشكلة القديمة:

- **المطور:** "الكود جاهز — ألقوه للعمليات"
- **العمليات:** "الكود لا يشتغل — أرجعوه للتطوير"
- **النتيجة:** كل فريق يلوم الآخر. المنتج يتأخر.

حل DevOps: **"أنت تبنيها، أنت تشغلها."** نفس الفريق يطور وينشر ويدير.

## إطار CALMS

| العنصر          | المعنى                   | مثال CloudNova                |
| --------------- | ------------------------ | ----------------------------- |
| **C**ulture     | مسؤولية مشتركة، لا لوم   | المطورون تحت الطلب لخدماتهم   |
| **A**utomation  | CI/CD، IaC، اختبارات     | GitHub Actions + Terraform    |
| **L**ean        | دفعات صغيرة، إزالة الهدر | فروع قصيرة، نشر متكرر         |
| **M**easurement | مقاييس، مراقبة           | DORA metrics                  |
| **S**haring     | توثيق، نقل معرفة         | Postmortems، Pair Programming |

## DORA Metrics — كيف تقيس؟

| المقياس                 | النخبة    | المتوسط | المتدني |
| ----------------------- | --------- | ------- | ------- |
| **تكرار النشر**         | حسب الطلب | أسبوعي  | شهري    |
| **Lead Time**           | < ساعة    | أسبوع   | > شهر   |
| **MTTR**                | < ساعة    | يوم     | > شهر   |
| **Change Failure Rate** | < ٥٪      | ١٠-١٥٪  | > ١٥٪   |

### هدف CloudNova: النخبة

```python
# لوحة قيادة DORA
metrics = {

    "lead_time_hours": 2.5,             # ✅ Elite

```

## أنماط DevOps الخاطئة

| النمط                    | المشكلة                   | الحل                                |
| ------------------------ | ------------------------- | ----------------------------------- |
| "فريق DevOps منفصل"      | أنشأت صومعة جديدة         | ادمج مهارات العمليات في فرق التطوير |
| "اشترينا Kubernetes"     | الخلط بين الأداة والفلسفة | DevOps ثقافة، Kubernetes أداة       |
| "نرمي للعمليات"          | لا ملكية مشتركة           | المطورون يمتلكون الإنتاج            |
| "التسليم السريع بأي ثمن" | تجاهل الجودة              | السرعة + الاستقرار معاً             |

## ثقافة Postmortem الـ Blameless

عندما يحدث عطل — لا تبحث عن **من** المخطئ. ابحث عن **ماذا** حدث وكيف نمنعه.

### هيكل Postmortem:

```markdown
# Postmortem: API Outage — ١٥ يناير ٢٠٢٤

## 📅 الجدول الزمني

- ١٤:٣٠: نشر v2.3 للإنتاج
- ١٤:٣٢: أول 500 Error
- ١٤:٣٣: PagerDuty ينبه
- ١٤:٣٧: تراجع فوري (rollback)
- ١٤:٤٠: الخدمة عادت — ٨ دقائق تعطل

## 📊 التأثير

- ١٢٧ عميلاً تأثروا
- ٠ طلبات فُقدت (كلها أعيدت محاولتها)
- ٠ بيانات فُقدت

## 🔍 السبب الجذري

- ConfigMap غيّر DATABASE_URL
- قيمة جديدة كانت خاطئة
- لم يتم اختبار التغيير في staging

## ✅ Action Items

1. [ ] أضف اختبار تكامل يتحقق من صحة ConfigMap (محمد — ٢٢ يناير)
2. [ ] أضف canary deployment ١٠٪ أولاً (أحمد — ٢٩ يناير)
3. [ ] وثق عملية تغيير ConfigMap (سارة — ٢٢ يناير)
```

## SRE — Site Reliability Engineering

| المفهوم          | المعنى                                          |
| ---------------- | ----------------------------------------------- |
| **SLO**          | Service Level Objective — هدفك: 99.9% توفر      |
| **SLI**          | Service Level Indicator — القياس الفعلي: 99.95% |
| **SLA**          | Service Level Agreement — ما وعدت به العميل     |
| **Error Budget** | 1 - SLO = مسموح لك بـ 0.1% تعطل                 |

```
SLO = 99.9% → Error Budget = 0.1%
= 43 دقيقة تعطل شهرياً مسموح بها
إذا استهلكت الميزانية → أوقف النشرات الجديدة — ركز على الاستقرار
```
