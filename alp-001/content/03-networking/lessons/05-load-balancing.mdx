## Simple Explanation

A load balancer is like a restaurant host: when you arrive, the host checks which waiter has the fewest tables and sends you there. If a waiter calls in sick (health probe fails), the host stops sending them customers.

<Analogy>

**The Supermarket Analogy**

- **Layer 4 (Azure LB):** Multiple checkout lanes. The manager sees lane 3 is empty and directs you there. They don't know or care what's in your cart.
- **Layer 7 (App Gateway):** A specialty store where the manager looks at your items and says: "Alcohol? Lane 5 is the only one authorized." They inspect your cart (HTTP headers) and route accordingly.

L4 is faster. L7 is smarter.

</Analogy>

### Health Probes: Design Matters

```
Bad probe:  TCP port 80 open? → "Healthy!" (even if app returns 500s)
Good probe: GET /health → returns 200 + checks DB connection → "Healthy!"

Great probe: GET /health/deep → returns 200, checks:
  1. Database: connected ✓
  2. Redis: reachable ✓
  3. Queue: not backlogged ✓
  4. Disk: < 80% full ✓
```

<ProductionNote>

### CloudNova: The Health Probe That Lied

**Incident #IN-228 — "API returns 500s but load balancer says healthy"**

The inventory API's health endpoint returned `200 OK` just by checking if the process was running. When the database became unreachable (network partition), the health check still passed — the process was alive, just useless.

**Root cause:** Health probes didn't validate actual dependencies.

**Fix:**

```python
@app.get("/health/deep")
async def deep_health():
    checks = {
        "database": await check_db(),
        "redis": await check_redis(),
        "disk": check_disk_space(),
    }
    all_healthy = all(checks.values())
    status_code = 200 if all_healthy else 503
    return {"status": "healthy" if all_healthy else "degraded", "checks": checks}, status_code
```

After fix: database partition → health returns 503 → LB removes backend → traffic routes to healthy region.

</ProductionNote>

## Professional Explanation

### Load Balancing Algorithms

| Algorithm             | How it works                                               | Best for                 |
| --------------------- | ---------------------------------------------------------- | ------------------------ |
| **Round Robin**       | Each request to next server in list                        | Equal-capacity backends  |
| **Least Connections** | Server with fewest active connections                      | Mixed-duration requests  |
| **IP Hash (5-tuple)** | Hash of src IP, dst IP, src port, dst port, protocol       | Consistent routing (NAT) |
| **Weighted**          | Percentage split (80/20)                                   | Canary deployments       |
| **Path-based**        | `/api/*` → pool A, `/static/*` → pool B                    | Microservices routing    |
| **Host-based**        | `api.cloudnova.io` → pool A, `admin.cloudnova.io` → pool B | Multi-tenant             |

### Session Affinity (Sticky Sessions)

When a user must stay on the same backend (e.g., websocket, server-side session state):

```
Without affinity: User → LB → Server A  (request 1)
                  User → LB → Server B  (request 2)  ← Session lost!

With affinity:    User → LB → Server A  (request 1)
                  User → LB → Server A  (request 2)  ← Same server
```

<Warning>
  Session affinity creates a scaling problem: if Server A goes down, all its users lose sessions.
  **Prefer stateless backends** — store sessions in Redis/CosmosDB instead.
</Warning>

**Key design decisions:**

1. Front Door at edge (anycast) → lowest latency for global users
2. App Gateway per region → L7 routing, WAF at regional level
3. Azure LB for VMSS → L4 pass-through for non-HTTP workloads
4. CDN for static assets → offload from application

## Hands-On Exercise

<Exercise>

### Lab: Configure Azure Load Balancer

```bash
# Create public LB
az network lb create \
  --resource-group cloudnova-lab \
  --name cn-public-lb \
  --sku Standard \
  --frontend-ip-name frontend \
  --public-ip-address cn-lb-pip

# Create backend pool
az network lb address-pool create \
  --resource-group cloudnova-lab \
  --lb-name cn-public-lb \
  --name web-pool

# Create health probe
az network lb probe create \
  --resource-group cloudnova-lab \
  --lb-name cn-public-lb \
  --name http-health \
  --protocol Http \
  --port 80 \
  --path /health

# Create load balancing rule
az network lb rule create \
  --resource-group cloudnova-lab \
  --lb-name cn-public-lb \
  --name http-rule \
  --protocol Tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name frontend \
  --backend-pool-name web-pool \
  --probe-name http-health
```

</Exercise>

<InterviewQuestion>

**Q:** "How do you handle 'thundering herd' on service restart?"
**A:** "When a service restarts, all clients simultaneously reconnect, flooding it. Solutions: 1) Jitter in health check intervals on backends, 2) Gradual warmup — LB sends increasing traffic percentage over 60-120 seconds, 3) Connection pooling — clients reuse connections, reducing reconnect storms. App Gateway v2 supports connection draining and gradual warmup."

**Q:** "What's the difference between Azure Front Door and Traffic Manager?"
**A:** "Traffic Manager is DNS-only — it returns different IPs based on routing rules, but all traffic goes directly to the backend (no data-plane). Front Door is a global reverse proxy with anycast — traffic flows through Microsoft's edge network, providing WAF, TLS termination, URL routing, and caching at the edge. Use Traffic Manager for simple global failover. Use Front Door for global acceleration + security."

</InterviewQuestion>
