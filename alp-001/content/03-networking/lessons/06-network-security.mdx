## Core Explanation

### NSG Rules: Priority and Evaluation

NSG rules are evaluated in **priority order** (lowest number wins). The last rule (65500) is always the default "Deny All."

<Mermaid>
graph TD
    Packet["📦 Incoming Packet<br/>src: 10.0.2.5:443 → dst: 10.0.1.10:80"]
    
    Packet --> Rule100{"Rule 100<br/>Allow 10.0.2.0/24:80?"}
    Rule100 -->|"✅ Match! ALLOW"| Pass["Packet passes"]
    Rule100 -->|No match| Rule200{"Rule 200<br/>Deny 10.0.2.0/24:22?"}
    Rule200 -->|No match| Rule300{"Rule 300<br/>Allow 10.0.1.0/24:80?"}
    Rule300 -->|"❌ No match (wrong src)"| Rule65500{"Default: DenyAllInbound"}
    Rule65500 -->|"❌ Deny"| Drop["Packet dropped 🔒"]

    style Pass fill:#22c55e,color:#000
    style Drop fill:#ef4444,color:#fff
</Mermaid>

**Key NSG facts:**
- Rules are **stateful**: allow inbound → return traffic automatically allowed
- Priority: 100-4096 (lower = higher priority)
- Service Tags: `Internet`, `AzureLoadBalancer`, `VirtualNetwork` are built-in
- Application Security Groups (ASGs): group VMs by role, not IP

<Example>

```bash
# Create NSG with least-privilege rules
az network nsg create --name web-nsg --resource-group cloudnova-prod

# Rule 100: Allow HTTP from anywhere
az network nsg rule create --name AllowHTTP --nsg-name web-nsg \
  --priority 100 --direction Inbound \
  --source-address-prefixes Internet \
  --destination-port-ranges 80 443 --protocol Tcp --access Allow

# Rule 200: Allow SSH ONLY from bastion subnet
az network nsg rule create --name AllowSSH-from-Bastion --nsg-name web-nsg \
  --priority 200 --direction Inbound \
  --source-address-prefixes 10.0.0.32/27 \
  --destination-port-ranges 22 --protocol Tcp --access Allow

# Rule 300: Deny all other inbound (explicit — default already does this)
az network nsg rule create --name DenyAllInbound --nsg-name web-nsg \
  --priority 4000 --direction Inbound \
  --source-address-prefixes '*' \
  --destination-port-ranges '*' --protocol '*' --access Deny
```

</Example>

### NSG vs Azure Firewall: When to Use Which

| Concern | NSG | Azure Firewall |
|---------|-----|---------------|
| **Scope** | Subnet or NIC | Hub VNet (centralized) |
| **Filtering** | IP, port, protocol | IP, port, FQDN, threat intel |
| **Logging** | NSG flow logs | Firewall logs + Log Analytics |
| **Cost** | Free | $900+/month (Standard) |
| **State** | Stateful (5-tuple) | Stateful + application-level |
| **Best for** | Granular subnet rules | Central egress/ingress control |

<SecurityNote>

**Rule of thumb:** NSGs for east-west (subnet-to-subnet) and basic filtering. Azure Firewall for north-south (Internet ↔ VNet), FQDN filtering, and centralized logging. Use both — they complement, not replace.

</SecurityNote>

### CloudNova: The RDP Incident

**Incident #IN-601 — "Unauthorized RDP access detected"**

A developer opened RDP (port 3389) to `0.0.0.0/0` on a test VM "just for 10 minutes." They forgot to close it. Within 6 hours, Azure Security Center detected brute-force attempts from 47 different IP addresses in 5 countries.

**Fix implemented:**
1. **Azure Policy** to deny NSG rules with source `0.0.0.0/0` and destination ports 22/3389
2. **Just-in-Time (JIT) VM Access** — request temporary access with justification
3. **Bastion** for all VM access — no public RDP/SSH ever
4. **Automated remediation** — Logic App that removes allow-all rules within 5 minutes

</ProductionNote>

## Architect Explanation

### Microsegmentation

Instead of one big "internal network," every workload gets its own security boundary:

```
❌ Old way: One subnet, all VMs can talk to each other
   ┌──────────────────────────────────┐
   │ Web VM → App VM → DB VM  (all open internally) │
   └──────────────────────────────────┘

✅ Microsegmentation:
   ┌────┐  ┌──────┐  ┌────┐
   │Web │→ │ App  │→ │ DB │  (only specific ports between specific tiers)
   └────┘  └──────┘  └────┘
   No Web→DB direct, No App→Internet, No external→App
```

**Implementation in Azure:**
1. NSGs at subnet level (broad rules)
2. ASGs for application-level groups
3. Azure Policy to enforce NSG on every subnet
4. Network Watcher to verify actual flows match expected rules
