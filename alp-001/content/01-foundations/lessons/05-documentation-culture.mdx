# Documentation Culture

:::level simple

**Documentation is like a map.** When you first explore a city, you need a map. After living there for years, you don't need it anymore — but every new person does.

Engineers often skip documentation because they already know their own systems. They forget that every new team member is exploring the city for the first time, without a map.

**The rule:** If a new engineer can't understand your system from the docs in their first week, the docs have failed. Not the engineer.

:::

:::level core

## Docs-as-Code: Documentation Lives in Git

Documentation-as-code means treating docs exactly like software:

```mermaid
graph LR
    A[Write Docs in Markdown] --> B[Store in Git]
    B --> C[Review via Pull Request]
    C --> D[CI/CD Validates]
    D --> E[Auto-Deploy to Website]

    style A fill:#0ea5e9,stroke:#0284c7,color:#fff
    style C fill:#f59e0b,stroke:#d97706,color:#000
    style E fill:#10b981,stroke:#059669,color:#fff
```

At CloudNova, every PR that changes infrastructure must either:

- Include updated documentation, or
- Explicitly state why docs don't need updating

This single rule eliminated 80% of stale documentation in 3 months.

:::

## Key Takeaways

- **Docs-as-code** means docs in git, reviewed in PRs, deployed via CI/CD — just like code.
- **Living documentation** is updated alongside the system; stale docs are worse than no docs.
- **Your future self** is the primary audience. Write for the person debugging this at 3 AM.
