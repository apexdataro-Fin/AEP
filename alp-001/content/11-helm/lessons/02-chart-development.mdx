## Building Production Charts

### Simple

A Helm chart is a directory with a predictable structure:

```
mychart/
├── Chart.yaml          # Chart metadata (name, version, description)
├── values.yaml         # Default configuration values
├── charts/             # Sub-charts (dependencies)
├── templates/          # Kubernetes manifest templates
│   ├── NOTES.txt       # Post-install help text
│   ├── _helpers.tpl    # Reusable template helpers
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── hpa.yaml
│   └── tests/          # Helm test definitions
│       └── test-connection.yaml
└── .helmignore         # Files to exclude from packaging
```

Start with `helm create mychart` — it generates this structure following community best practices.

### Core

**The `_helpers.tpl` file** is your chart's shared template library. Define named templates once, reuse everywhere:

```yaml
{{/*
Create a default fully qualified app name.
*/}}
{{- define "mychart.fullname" -}}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- $name := default .Chart.Name .Values.nameOverride }}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" }}
{{- end }}
{{- end }}

{{/*
Common labels
*/}}
{{- define "mychart.labels" -}}
helm.sh/chart: {{ include "mychart.chart" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}
```

Use named templates: `{{ include "mychart.fullname" . }}`

### Professional

**Conditional template blocks** using `if/else` with values:

```yaml
# templates/ingress.yaml
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ include "mychart.fullname" . }}
  {{- with .Values.ingress.annotations }}
  annotations:
    {{- toYaml . | nindent 4 }}
  {{- end }}
spec:
  {{- if .Values.ingress.tls }}
  tls:
    - hosts:
        - {{ .Values.ingress.host }}
      secretName: {{ .Values.ingress.tlsSecretName }}
  {{- end }}
  rules:
    - host: {{ .Values.ingress.host }}
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: {{ include "mychart.fullname" . }}
                port:
                  number: {{ .Values.service.port }}
{{- end }}
```

**Range loops** for iterating over lists:

```yaml
# templates/configmap.yaml
data:
  {{- range .Values.env }}
  {{ .name }}: {{ .value | quote }}
  {{- end }}
```

**With blocks** for scoping contexts:

```yaml
{{- with .Values.resources }}
resources:
  {{- toYaml . | nindent 2 }}
{{- end }}
```

### Production

**Helm hooks** give you precise lifecycle control:

| Hook | Execution Point |
|------|----------------|
| `pre-install` | After templates rendered, before resources created |
| `post-install` | After all resources are loaded |
| `pre-delete` | Before any resources are deleted |
| `post-delete` | After all resources deleted |
| `pre-upgrade` | After templates rendered, before upgrade |
| `post-upgrade` | After all resources upgraded |
| `pre-rollback` | Before rollback executes |
| `post-rollback` | After rollback completes |
| `test` | When `helm test` is invoked |

Hooks are implemented by adding annotations to any Kubernetes resource:

```yaml
# templates/db-migration.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ include "mychart.fullname" . }}-db-migrate
  annotations:
    "helm.sh/hook": post-install,post-upgrade
    "helm.sh/hook-weight": "5"
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          command: ["python", "manage.py", "migrate"]
```

**Hook deletion policies:**
- `before-hook-creation` — delete previous hook before running new one
- `hook-succeeded` — delete after success
- `hook-failed` — keep on failure for debugging

### Architect

**Chart dependencies** enable modular composition:

```yaml
# Chart.yaml
dependencies:
  - name: redis
    version: "18.x.x"
    repository: "oci://myregistry.azurecr.io/helm"
    condition: redis.enabled
  - name: postgresql
    version: "15.x.x"
    repository: "oci://myregistry.azurecr.io/helm"
    condition: postgresql.enabled
    tags:
      - database
```

**Sub-chart values override** via the parent values.yaml:

```yaml
# Parent values.yaml
redis:
  enabled: true
  architecture: replication
  auth:
    enabled: true
    password: ""  # Set via external secret

postgresql:
  enabled: true
  primary:
    persistence:
      size: 50Gi
```

## Hands-On Exercise

```bash
# 1. Scaffold a new chart
helm create flask-api

# 2. Examine the structure
tree flask-api/

# 3. Customize values for your app
# Edit flask-api/values.yaml
# - Change image.repository to your container registry
# - Set service.type to ClusterIP
# - Enable ingress

# 4. Add a ConfigMap template
cat > flask-api/templates/configmap.yaml << 'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "flask-api.fullname" . }}-config
data:
  FLASK_ENV: {{ .Values.environment | quote }}
  LOG_LEVEL: {{ .Values.logLevel | default "INFO" | quote }}
EOF

# 5. Lint the chart
helm lint flask-api/

# 6. Dry-run to see rendered templates
helm install --dry-run --debug test-release flask-api/

# 7. Package the chart
helm package flask-api/

# 8. Create an index for a local repo
helm repo index . --url https://charts.example.com
```

## Flashcards

**Q:** What command scaffolds a new Helm chart with best practices?
**A:** `helm create <chart-name>`

**Q:** What file holds reusable template functions shared across all templates?
**A:** `templates/_helpers.tpl`

**Q:** What annotation triggers a Helm hook?
**A:** `"helm.sh/hook": post-install,post-upgrade`

**A:** `helm lint <chart-dir>` and `helm install --dry-run --debug`
