# Service Catalog & Developer Self-Service

<Definition>

The **Service Catalog** is the heart of an Internal Developer Platform — a unified registry of every service, who owns it, its dependencies, and its operational status. **Scaffolding** lets developers create new services with best practices baked into templates.

</Definition>

---

## 🎯 Learning Objectives

- Design a service catalog that teams actually use (not abandon)
- Build Backstage software templates for self-service
- Measure platform adoption with developer experience metrics

---

## 🔥 Core Explanation

### What Makes a Good Service Catalog?

| Must Have | Why |
|-----------|-----|
| **Ownership** | Who's on-call at 3 AM? |
| **Dependencies** | If Database-A goes down, what breaks? |
| **API docs** | How do I call this service? |
| **CI/CD status** | Is the latest build green? |
| **Deployment info** | What version is in production? |
| **Runbooks** | How do I respond to its alerts? |

<BestPractice>

**A catalog that relies on manual updates is a dead catalog.** CloudNova's catalog auto-discovers services from GitHub repos (looking for `catalog-info.yaml`), auto-updates CI/CD status from pipelines, and auto-links documentation from the repo. Zero manual maintenance.

</BestPractice>

## 🏗️ Professional Explanation

### Software Templates — Scaffolding Done Right

<CodeBlock language="yaml" title="Backstage Template: New Microservice">
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: cloudnova-microservice
  title: Create a CloudNova Microservice
spec:
  owner: platform-team
  type: service
  parameters:
    - title: Service Details
      required: [name, owner, language]
      properties:
        name:
          title: Service Name
          type: string
        owner:
          title: Owning Team
          type: string
          ui:field: OwnerPicker
        language:
          title: Language
          type: string
          enum: [python, go, typescript]
  steps:
    - action: fetch:template
      input:
        url: ./skeleton
    - action: publish:github
      input:
        repoUrl: github.com?owner=apexdataro-Fin&repo={{name}}
    - action: catalog:register
      input:
        repoContentsUrl: '{{steps.publish.output.repoContentsUrl}}'
</CodeBlock>

<ProductionNote>

**CloudNova's template produces a fully operational service in 5 minutes.** Developer fills in name, team, and language. The template generates: repo with CI/CD, Dockerfile, Kubernetes manifests, monitoring dashboards, and a Slack channel. Previously, this took 2 days of setup across 4 different tools.

</ProductionNote>

## 🧪 Active Recall

<Flashcard
  front="What's the most important field in a service catalog entry?"
  back="**Ownership** — who owns this service, who's on-call for it. Without clear ownership, a catalog is just a pretty list. With ownership, it's an operational tool."
/>

<Flashcard
  front="Why is auto-discovery critical for a service catalog?"
  back="Manual updates don't happen. Teams forget. The catalog becomes stale and untrusted. Auto-discovery from Git repos, CI/CD, and monitoring ensures the catalog reflects reality — not what someone remembered to update."
/>
